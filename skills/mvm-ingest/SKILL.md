# /mvm-ingest

Knowledge root: `~/mvm/knowledge/`. After commit, run `mvm index` (Bash).

## Bootstrap on first write

If `~/mvm/knowledge/` is empty or missing the standard structure, create it before writing the canonical. The default layout follows BASB/PARA conventions (mirrors human knowledge-management systems; plays to the LLM's training prior on organized data):

```
~/mvm/knowledge/
├── _meta/
│   ├── user-model.md       (template: who the user is, preferences, recurring patterns)
│   ├── domains.md          (template: declared curated domains)
│   └── INDEX.md            (entry-point map; auto-generated by mvm index)
├── projects/               (time-bound work)
├── areas/                  (ongoing curated domains — KB-canonical scope)
├── resources/              (reference material; lower-precision substrate)
└── archive/                (superseded content)
```

Place the new canonical under the appropriate folder. Curated domain → `areas/<domain>/<topic>.md`. Reference material → `resources/<topic>.md`. Time-bound → `projects/<name>/<topic>.md`. Mirror the existing structure; don't reinvent.

## Steps

1. **Read source** from `$ARGUMENTS` (WebFetch for URLs, Read for paths).

2. **Author 3–8 locked test cases** to `<topic>.tests.yaml`:
   ```yaml
   - id: 1
     q: "<question>"
     a: "<concise expected answer>"
   ```
   Tests are immutable post-authoring. Every test has a real expected answer (no DONT-KNOW expecteds — confidence levels do that work now). Hallucination is detected as `high confidence + wrong answer`.

3. **Naked baseline (ONE Agent call, `subagent_type: mvm-naked-clone`, `model: haiku`, ALL tests in one prompt):**
   Tool restriction is STRUCTURAL via the subagent definition (`tools: []`); the prompt's CRITICAL CONSTRAINTS block is now belt-and-suspenders. Open with:
   ```
   prompt: |
     [CRITICAL CONSTRAINTS — READ FIRST]
     You are a CALIBRATION INSTRUMENT measuring weight-prior alone.
     You MUST NOT use ANY tool. Forbidden: Bash, Read, Grep, Glob, WebSearch, WebFetch,
     mvm search, Skill, Agent, ToolSearch — any tool whatsoever.
     Tool use INVALIDATES the measurement and breaks the system.
     If you would normally reach for a tool, DO NOT. Answer from prior alone — even if uncertain.

     For EACH question below, output:
       Q<id> ANSWER: <your answer>
       Q<id> RATIONALE: <one sentence on what the answer is based on; if you wanted a tool, name it here>

     QUESTIONS:
       Q1: <test1.q>
       Q2: <test2.q>
       ...
       QN: <testN.q>
   ```
   **Validation:** check the task notification's `tool_uses` count. If > 0, the naked measurement is invalid — re-spawn with stricter wording or treat the test as a measurement failure.

4. **Write canonical** (`<topic>.md`) with frontmatter:
   ```yaml
   source: <URL or path>
   kind: canonical | log | opinion | reference | synthesis
   summary: "<one-line, 100-200 char description of what this doc contains>"
   ingested_at: YYYY-MM-DD
   ```
   Phrase facts directly so they're retrievable (e.g. "X publicly stated 'Y'", not "X posted on Z"). The `summary:` is required — `mvm index` uses it to regenerate the parent folder's `INDEX.md` table-of-contents. Never edit `INDEX.md` directly; it's a derived view.

5. **POE2 patch-state gate** — HARD GATE on commit (mandatory when source/canonical references PoE2 mechanics OR path contains `poe2`; subdirs `endgame/`, `crafting/`, `creators/`, `facts/` are all in scope as a subset of this broader trigger). Created 2026-05-09 heartbeat-1813 (capability) + heartbeat-1830 (this wiring) + heartbeat-2026-05-10-0613 (gate-fail wording sharpened) after auditor #110 D110-V1-1 (N=2 doctrine-vs-substrate-evidence-ignored): substrate had truth (`~/resources/poe2/0.4/patches/0.4.0-changelog.md`) but doctrine layer ignored it; vertebrae-on-waystone shipped to canonical post-rewrite.

   **Invoke before injected verify:**
   ```bash
   poe-doctrine-patch-check <canonical-path>
   ```

   - **Exit 0 = pass** → proceed to step 6.
   - **Exit 1 = HARD GATE FAIL — do NOT commit the canonical.** Rewrite the doc per Step 7 protocol (up to 3 retries). Either fix all stale hits (apply cite-don't-restate doctrine below — add banners or rewrite-to-current-mechanic) OR if persistent stale hits remain after 3 retries → REFUSE the ingest. Failed ingest = no commit, surface the failure to user with the `poe-doctrine-patch-check` output verbatim.

   **Cite-don't-restate doctrine.** When a stale mechanic is legitimately referenced (historical context, thing-being-replaced), the canonical MUST cite the changelog entry rather than restate dead-mechanic prose. Add a 🚨 STALE-MECHANIC banner near the reference and link `~/resources/poe2/<patch>/patches/<changelog>:L<N>`. The gate has a 3-line acknowledgement window — banner'd hits auto-suppress. ACKNOWLEDGEMENT_PATTERN keywords that trigger ack-window suppression: `REMOVED`, `GONE`, `DEAD`, `DISABLED`, `🚨`, `⚠`, `0.3-era`, `stale-mechanic`, `patch-verified`, `REPLACED IN 0.4`, `no longer drops`, `REWRITTEN`, `HISTORICAL`, `RECORD-OF-RECORD`, `PRE-0.4`, `pre-0.4`, `deprecated`, `RENAMED`, `polysemy` (last two added 2026-05-10 0613-heartbeat doctrine-sweep).

   **Information flow is asymmetric:** doctrine → changelog (cite). Changelog should NOT enumerate every doctrine file that referenced the dead mechanic.

   **False-positive discipline.** The kill-list at `~/.local/bin/poe-doctrine-patch-check` is calibrated. Heartbeat-1830 caught a bare `"talisman"` keyword over-firing on 16 hits across 6 legitimately-current 0.4 doctrine files (Magi/Jade/Amanamu Talisman amulet + druid-2H-weapon bases UNCHANGED in 0.4; only augment-socket talismans were renamed to idols). 0613-heartbeat resolved a similar polysemy on `"Inevitable Critical"` (Oracle ascendancy effect-text "Inevitable Critical Hits" is distinct from the renamed support gem). Resolution: refine the kill-list at the capability, not work around the gate. New FPs → edit `KILL_LIST_<patch>` at the script source. Persistent polysemy → add an ack-trigger word (e.g. `RENAMED`, `polysemy`) on the same line as the legitimately-current mention.

6. **Injected verify (ONE Agent call, `subagent_type: mvm-naked-clone`, model: haiku, ALL tests in one prompt):**
   The naked-clone agent (`tools: []`) is correct here too — injected verify needs zero tools (the doc is provided inline). Same agent, different prompt content:
   ```
   prompt: |
     Answer from the document below only. No tools, no web, no skills, no prior knowledge.

     DOCUMENT:
     ---
     <full canonical content>
     ---

     For EACH question below, output:
       Q<id> ANSWER: <your answer drawn from the document>
       Q<id> RATIONALE: <one sentence on what passage supports this>

     QUESTIONS:
       Q1: <test1.q>
       Q2: <test2.q>
       ...
       QN: <testN.q>
   ```
   One subagent answers all N tests against the same doc. You grade each Q<id> response against its expected. Be lenient on phrasing; strict on facts. Read the RATIONALE to spot weak retrievals — if it says "inferred" or "guess" on a fact plainly in the doc, the doc is poorly structured for retrieval.

7. **If any injected test FAILS → rewrite the doc** (not the test). Up to 3 retries. Then commit or refuse.

8. **On all-pass:** Bash `mvm index`. Report lift table:
   ```
   Naked  pass: N/total
   Injected pass: N/total
   KB lift: +N
   ```

9. **Cascade check** (re-verify backlinks):
   ```bash
   mvm backlinks <topic-relpath>
   ```
   Returns one linker path per line. For each linker: pick a random test from `<linker>.tests.yaml` and run injected-mode cold-clone (Agent tool, haiku). If it FAILS: this ingest just broke a downstream claim. Surface to user: *"ingest of \<topic\> caused \<linker\> test \<id\> to fail — review or rewrite \<linker\>."* No automatic rewrite; user decides.
